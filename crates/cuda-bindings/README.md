# cuda-bindings

Raw FFI bindings to the CUDA driver API, generated by [bindgen](https://crates.io/crates/bindgen) from `wrapper.h` (which includes `cuda.h`).

## What this crate provides

- All types, constants, and function declarations from `cuda.h` (e.g., `cuInit`, `cuLaunchKernel`, `CUdeviceptr`, `CUstream`)
- A `cuda_toolkit_dir()` helper that resolves the toolkit root at runtime

## Build requirements

- CUDA Toolkit installed (headers in `<toolkit>/include/cuda.h` or `<toolkit>/targets/<dir>/include/cuda.h`)
- `clang` available for bindgen (typically bundled with LLVM)

The build script locates the toolkit via the `CUDA_TOOLKIT_PATH` environment variable, then `CUDA_HOME`, falling back to `/usr/local/cuda`. Headers are discovered in `include/` or, for redistributable layouts without a top-level `include/`, in the single `targets/<dir>/include/` matching the build target (`x86_64-linux` for x86_64 hosts, `sbsa-linux` for aarch64 servers). When `cuda.h` is not found, the build fails with one message listing the probed paths instead of a bindgen panic. The script links against `libcuda.so` (the driver stub) and searches `lib64/` plus the matching target's `lib/` and `lib/stubs/` for the native library.

## Environment variables

| Variable            | Purpose                                    | Default            |
|---------------------|--------------------------------------------|--------------------|
| `CUDA_TOOLKIT_PATH` | Root of the CUDA Toolkit installation      | `/usr/local/cuda`  |
| `CUDA_HOME`         | Fallback when `CUDA_TOOLKIT_PATH` is unset | `/usr/local/cuda`  |

## Usage

This crate is not intended for direct use. Prefer the safe wrappers in `cuda-core` or the async layer in `cuda-async`.

```rust
// Low-level access (if you need it):
use cuda_bindings::{cuInit, cuDeviceGet, CUdevice};
```
